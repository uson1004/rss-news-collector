# 읽을게 4차 개발 보고서

작성 기준: 2026년 6월 22일

## 1. 개발 진행 개요

3차 개발에서는 뉴스레터 정기 발송을 위한 배치 처리와 운영 안정성 개선에 집중하였다.

4차 개발에서는 읽기 경험을 확장하기 위해 **AI 관찰 노트, URL 입력 개선, Supabase 연결 검증, 아이디어 레이더 화면 개편, 모바일 대응과 접근성 보강**을 진행하였다.

이번 단계는 단순히 기사를 요약하는 기능을 넘어, 사용자가 기사에서 변화 신호와 아이디어를 발견할 수 있도록 돕고 향후 외부 데이터 저장소로 전환하기 위한 기술 검증을 수행한 단계이다.

## 3. AI 관찰 노트 기능

기존 AI 요약은 원문을 짧게 압축하는 기능이었다. 4차 개발에서는 요약과 별도로 기사에서 읽을 가치와 아이디어 신호를 찾는 **AI 관찰 노트**를 추가하였다.

### 관찰 항목

| 항목 | 내용 |
| --- | --- |
| 왜 볼 만한가 | 사용자가 글을 읽을 필요를 빠르게 판단할 수 있는 근거 |
| 보이는 변화 | 기사에서 관찰되는 사회·생활·기술 변화 |
| 불편과 욕망 | 사용자의 문제, 요구, 제도와 현실의 간극 |
| 아이디어 힌트 | 제품·서비스·콘텐츠 아이디어로 발전시킬 수 있는 질문 |
| 주의할 점 | 근거가 약하거나 AI 해석에 주의해야 하는 부분 |

### 처리 방식

```text
[기사 제목·본문·출처·카테고리]
        ↓
[Claude API 분석 요청]
        ↓
[구조화된 JSON 응답]
        ↓
[항목별 개수와 문장 정리]
        ↓
[리더뷰 관찰 노트 패널 표시]
```

AI가 원문에 없는 내용을 사실처럼 단정하지 않도록 가능성, 신호, 힌트 중심의 표현을 사용하게 하였다. 추출할 신호가 부족한 경우에는 아이디어를 억지로 생성하지 않고 주의 문구를 표시한다.

### API

```http
POST /api/article-insight
Content-Type: application/json

{
  "title": "기사 제목",
  "text": "기사 본문",
  "source": "매체명",
  "category": "idea_radar"
}
```

응답은 관찰 항목별 문자열 배열과 사용 모델명을 포함한다.

## 4. 아이디어 레이더 중심 화면 개편

뉴스 읽기 화면을 일반적인 기사 목록에서 **아이디어 레이더** 성격이 드러나는 편집형 화면으로 보강하였다.

- 오늘의 날짜와 선택 카테고리를 보여주는 표지 영역 추가
- 첫 번째 기사를 대표 기사로 강조
- 이후 기사를 순번이 있는 목차 형태로 배치
- 기사 출처를 먼저 표시하여 원문 중심 읽기 원칙 강화
- AI는 원문을 대신하지 않고 읽을 이유와 아이디어 힌트를 제공하는 보조 기능으로 배치
- 뉴스 화면, URL 입력 화면, 리더뷰 화면의 역할 분리 유지

## 5. URL 입력 및 본문 추출 개선

### 프로토콜 없는 URL 지원

기존에는 사용자가 `https://`를 포함한 완전한 주소를 입력해야 했다. 현재는 다음과 같이 프로토콜을 생략한 주소도 자동으로 HTTPS 주소로 보정한다.

```text
example.com/article
→ https://example.com/article
```

HTTP와 HTTPS가 아닌 FTP 등의 지원하지 않는 프로토콜은 입력 검증 단계에서 거부한다.

### RSS fallback 유지 및 보강

- 기사 페이지 직접 추출이 실패하면 공식 RSS 후보를 탐색
- Medium, Google Blog 계열의 피드 주소 규칙 지원
- RSS 항목의 URL, GUID, ID를 정규화하여 원문과 비교
- RSS 본문에서 이미지, 스크립트, iframe 등 불필요한 요소 제거
- 직접 접근 차단과 RSS 검색 실패 원인을 구분해 안내

## 6. Supabase 연동 검증

2차 보고서에서 계획한 외부 DB 전환을 준비하기 위해 Supabase JavaScript 클라이언트를 추가하였다.

현재 구현 범위는 다음과 같다.

- `VITE_SUPABASE_URL` 환경변수 사용
- `VITE_SUPABASE_PUBLISHABLE_KEY` 환경변수 사용
- Supabase 클라이언트 생성
- 공개 `todos` 테이블의 `id`, `name` 조회
- 로딩, 오류, 빈 목록, 조회 성공 상태를 뉴스 화면에 표시

이번 연동은 **Supabase 연결과 조회 동작을 확인하기 위한 초기 검증 단계**이다. 뉴스레터 카테고리, 구독 정보, 발송 기록은 아직 SQLite에 저장되며 사용자 인증도 구현되지 않았다.

## 7. 읽기 설정과 접근성·모바일 보강

### 읽기 설정 저장

글자 크기, 줄 간격, 문단 간격, 본문 폭, 화면 테마는 기존과 같이 브라우저 `localStorage`에 저장된다. 페이지를 다시 열어도 같은 브라우저에서는 설정을 유지할 수 있다.

다만 사용자 계정과 연결된 저장 방식은 아니므로 다른 기기나 브라우저와 설정이 동기화되지는 않는다.

### 접근성 관련 보강

- 주요 화면과 패널에 `aria-label`, `aria-labelledby` 적용
- 로딩 상태에 `aria-live` 적용
- 오류 상태에 `role="alert"` 적용
- 카테고리 선택 영역에 tablist 역할 적용
- 모달에 dialog 역할과 `aria-modal` 적용
- 읽기 조절 버튼에 동작을 설명하는 접근성 이름 적용

### 모바일 반응형 구성

760px 이하 화면에서 다음 요소가 한 열로 재배치된다.

- 상단 내비게이션
- URL 입력 폼
- RSS 기사 목록과 대표 기사
- 읽기 도구 모음
- TTS, AI 요약, AI 관찰 노트 패널
- 카테고리 추가 버튼과 모달

버튼을 화면 너비에 맞추고 기사 목록의 읽기 버튼을 별도 행으로 배치하여 작은 화면에서도 조작하기 쉽게 하였다.

## 8. 테스트 및 검증

### AI 관찰 노트 테스트

- 코드 블록이 포함된 JSON 응답 파싱
- 항목별 관찰 결과 변환
- 유효한 신호가 없을 때 주의 문구 추가
- Claude API 키가 없을 때 오류 반환

### URL 입력 테스트

- 프로토콜 없는 URL을 HTTPS로 변환
- 지원하지 않는 프로토콜 입력 거부

### 전체 검증 결과

- 백엔드 단위 테스트 12건 통과
- Python 주요 모듈 문법 검사 통과
- Vite 프로덕션 빌드 통과

외부 Claude API, DeepL API, Supabase, SMTP 서버와의 실제 연동은 각 서비스의 환경변수와 네트워크 설정이 필요하므로 로컬 자동 테스트 범위에는 포함하지 않았다.

## 10. 현재 제약 사항

- AI 관찰 노트와 요약에는 Claude API 키가 필요함
- 번역에는 DeepL API 키가 필요함
- Supabase 환경변수가 없으면 현재 프론트엔드 초기화 단계에서 오류가 발생할 수 있음
- Supabase는 공개 `todos` 조회에만 사용되며 서비스 핵심 데이터는 SQLite에 남아 있음
- 사용자 인증과 권한별 데이터 접근 제어가 없음
- 접근성 속성은 보강했지만 키보드 포커스 순서, 스크린리더, 색 대비에 대한 자동·수동 종합 검증이 추가로 필요함
- 카테고리 tablist의 tab 역할·선택 상태와 모달 포커스 고정·복원 같은 세부 키보드 패턴은 아직 구현되지 않음
- RSS와 본문 추출 결과는 외부 사이트의 차단 정책과 피드 품질에 영향을 받음
- 실제 모바일 기기와 여러 브라우저에 대한 교차 검증이 필요함

## 13. 결론

4차 개발에서는 읽기 중심 서비스의 방향을 강화하기 위해 AI 관찰 노트와 아이디어 레이더 화면을 추가하고, URL 입력 편의성과 모바일·접근성 요소를 보강하였다.

또한 Supabase 클라이언트와 공개 테이블 조회를 연결하여 외부 데이터 저장소 전환 가능성을 확인하였다. 다만 이는 전체 저장소 이전이 아니라 연결 검증 단계이며, 핵심 데이터는 여전히 SQLite에 저장된다.

현재 프로젝트는 기사 탐색, 본문 추출, 읽기 설정, AI 요약·관찰·번역, TTS, 사용자화 RSS 카테고리, 뉴스레터 구독과 정기 발송까지 지원한다. 다음 단계에서는 사용자 계정, 북마크, 읽기 기록, Supabase 기반 영구 저장, 배포와 접근성 검증을 중심으로 완성도를 높일 필요가 있다.
