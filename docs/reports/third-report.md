# 읽을게 3차 개발 보고서

작성 기준: 2026년 6월 22일

## 1. 개발 진행 개요

2차 보고서에서는 뉴스레터 구독, 미리보기, 테스트 발송 기능까지 구현되었지만, 정해진 주기에 따라 구독자에게 자동으로 발송하는 기능은 아직 구현되지 않은 상태로 정리하였다.

3차 개발에서는 이 미반영 항목을 중심으로 **주간 뉴스레터 정기 발송 구조, 중복 발송 방지, 발송 결과 기록, 관리자용 실행 API와 배치 스크립트**를 추가하였다.

이번 단계의 핵심은 뉴스레터를 단순히 구독하고 시험 발송하는 기능에서 벗어나, 외부 스케줄러에서 호출할 수 있는 배치 발송 경로로 확장한 것이다.

## 2. 2차 보고서 대비 변경사항

| 2차 보고서 내용 | 3차 개발 반영 결과 |
| --- | --- |
| 자동 정기 발송 스케줄러 미구현 | 주간 발송 대상을 처리하는 배치 로직과 실행 스크립트 구현 |
| 뉴스레터는 구독/테스트 발송 중심 | 구독자 목록 조회, 발송 대상 선별, 실제 SMTP 발송까지 연결 |
| 동일 구독자 중복 발송 방지 구조 없음 | ISO 주차 단위 발송 기록과 고유 제약조건으로 중복 발송 방지 |

| 운영 환경에서 실행할 관리 기능 부족 | 관리자 비밀값을 사용하는 배치 실행 API 추가 |

## 3. 주요 구현 기능

### 주간 뉴스레터 발송 대상 선별

- 활성 상태이며 발송 주기가 `weekly`인 구독만 조회
- UTC 기준 ISO 주차를 `YYYY-Www` 형식으로 계산
- 같은 구독자에게 같은 주차의 발송 기록이 있으면 대상에서 제외
- 한 번에 처리할 구독자 수를 제한할 수 있도록 `limit` 지원

### 중복 발송 방지

`newsletter_delivery_attempts` 테이블을 추가하여 구독별 발송 시도를 기록하도록 구성하였다.

| 저장 항목 | 용도 |
| --- | --- |
| subscription_id | 발송 대상 구독 식별 |
| category_id | 뉴스레터 카테고리 식별 |
| delivery_window | 발송 대상 주차 식별 |
| status | claimed, sent, failed 상태 기록 |
| claimed_at | 발송 처리 시작 시각 |
| sent_at | 발송 성공 시각 |
| failed_at | 발송 실패 시각 |
| error | 실패 원인 기록 |

`subscription_id`와 `delivery_window` 조합에 고유 제약조건을 적용하여, 동시에 여러 실행이 발생하더라도 같은 주차에 같은 구독을 중복 처리하지 않도록 하였다.

### 뉴스레터 배치 처리

`newsletter_delivery.py`에서 다음 순서로 발송을 처리한다.

```text
[주간 발송 대상 조회]
        ↓
[발송 시도 선점]
        ↓
[카테고리별 뉴스레터 생성]
        ↓
[SMTP 이메일 발송]
        ↓
[성공 또는 실패 결과 저장]
```

같은 카테고리를 구독한 사용자가 여러 명일 경우 뉴스레터 본문을 매번 다시 생성하지 않고, 한 번 만든 결과를 실행 중 캐시에 저장하여 재사용하도록 구성하였다.

### 배치 실행 스크립트

서버 디렉터리에서 다음 명령으로 발송 작업을 실행할 수 있다.

```bash
python deliver_newsletters.py --limit 100
```

실제 이메일을 보내거나 DB 상태를 변경하지 않고 대상만 확인하는 dry-run 기능도 제공한다.

```bash
python deliver_newsletters.py --dry-run --limit 10
```

### 관리자용 발송 API

서버 또는 배포 환경에서 배치 작업을 호출할 수 있도록 다음 API를 추가하였다.

```http
POST /api/newsletter/deliver-due
X-Newsletter-Admin-Secret: 설정한 관리자 비밀값
```

`NEWSLETTER_ADMIN_SECRET`이 없거나 요청 헤더의 값이 일치하지 않으면 요청을 거부한다. 이를 통해 일반 사용자가 전체 구독자 대상 발송 작업을 실행하지 못하도록 제한하였다.

### 구독 직후 첫 메일과 정기 발송 연계

구독 직후 첫 메일이 정상 발송되면 해당 주차의 발송 기록을 함께 저장한다. 따라서 같은 사용자가 그 주의 정기 배치에서 첫 메일과 동일한 뉴스레터를 다시 받지 않도록 개선하였다.

### 구독 해지 처리

구독별 해지 토큰을 조회하고 구독 상태를 `inactive`로 변경하는 API를 추가하였다.

```http
POST /api/newsletter/unsubscribe
```

비활성화된 구독은 이후 주간 발송 대상에서 제외된다.

## 4. 저장 구조 변경

기존 SQLite 저장소에 발송 상태 관리를 위한 테이블을 추가하였다.

```text
newsletter_subscriptions
 ├── 구독자 이메일
 ├── 카테고리
 ├── 발송 주기
 ├── 활성 상태
 ├── 해지 토큰
 └── 마지막 발송 시각

newsletter_delivery_attempts
 ├── 구독 ID
 ├── 카테고리 ID
 ├── 발송 주차
 ├── 처리 상태
 ├── 성공/실패 시각
 └── 오류 내용
```

이 구조를 통해 구독 정보와 개별 발송 시도를 분리하여 관리할 수 있게 되었다.

## 5. 개발 중 해결한 문제

| 문제 | 해결 방법 |
| --- | --- |
| 배치가 여러 번 실행될 때 중복 이메일이 발송될 수 있음 | 구독 ID와 발송 주차의 고유 제약조건 적용 |
| 동일 카테고리 뉴스레터를 구독자마다 다시 생성함 | 실행 단위 카테고리별 digest 캐시 적용 |
| SMTP 오류가 발생하면 처리 결과를 알기 어려움 | 실패 상태, 실패 시각, 오류 메시지를 저장 |
| 실제 발송 전 대상을 확인하기 어려움 | 상태 변경이 없는 dry-run 모드 추가 |
| 외부 환경에서 배치를 안전하게 호출하기 어려움 | 관리자 비밀값 검증 API 추가 |
| 구독 직후 첫 메일과 정기 메일이 중복될 수 있음 | 첫 메일 성공 시 현재 주차 발송 완료로 기록 |

## 6. 테스트 및 검증

뉴스레터 배치 기능에 대해 다음 항목을 자동 테스트로 확인하였다.

- 기본 카테고리 구독 생성
- UTC ISO 주차 계산
- 활성 구독과 비활성 구독 필터링
- 발송 시도 선점과 중복 처리 방지
- 같은 주차 재발송 차단
- 카테고리별 digest 재사용
- dry-run 실행 시 DB 상태가 변경되지 않는지 확인
- SMTP 실패 시 실패 상태가 기록되는지 확인
- 성공한 경우에만 마지막 발송 시각이 갱신되는지 확인

현재 전체 백엔드 테스트 12건이 통과하였으며, 프론트엔드 프로덕션 빌드도 정상 완료되었다. 실제 SMTP 서버와 배포 환경의 cron을 연결한 실서비스 발송은 자동 테스트 범위에 포함하지 않았다.

## 7. 현재 제약 사항

- 애플리케이션 내부에서 시간을 감시하는 스케줄러는 없으며 cron, 배포 플랫폼 스케줄러 등 외부 실행 환경이 필요함
- 현재 정기 발송 주기는 주간(`weekly`)만 지원함
- 실패한 발송은 기록되지만 자동 재시도하지 않음
- 실제 이메일 발송에는 SMTP 환경변수 설정이 필요함
- 저장소는 여전히 SQLite이며 여러 서버 인스턴스를 사용하는 환경에는 적합하지 않음
- 같은 이메일과 카테고리의 중복 구독을 막는 제약조건은 없어 반복 가입 시 여러 건의 구독이 생성될 수 있음
- 구독 해지 API는 구현되었지만 뉴스레터 본문의 사용자용 해지 링크 연결은 추가 작업이 필요함
- 관리자 API는 비밀값 비교 방식이며 운영 환경에서는 HTTPS와 안전한 비밀값 관리가 필요함

## 10. 결론

3차 개발에서는 2차 보고서에서 미구현 항목으로 남아 있던 뉴스레터 자동 정기 발송 기능을 운영 가능한 배치 구조로 확장하였다.

특히 주차 단위 중복 발송 방지, 성공·실패 기록, dry-run, 관리자 API를 추가하여 단순한 테스트 발송 기능보다 안정적으로 뉴스레터를 운영할 수 있는 기반을 마련하였다.

다만 실제 정해진 시간에 작업을 시작하는 역할은 외부 cron 또는 배포 플랫폼이 담당해야 하며, 전체 데이터 저장소의 외부 DB 전환과 사용자 계정 기능은 다음 개발 단계의 과제로 남아 있다.
